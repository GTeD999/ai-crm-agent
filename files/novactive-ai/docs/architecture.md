# Архитектура

## Высокоуровневый поток

```
Telegram User
    ↓
Telegram Bot API (webhook)
    ↓
FastAPI app (aiogram dispatcher)
    ↓
┌─────────────────────────────────────────────────┐
│  1. Получаем/создаём User в Supabase            │
│  2. Проверяем согласие на ПД (если нет — спрашиваем) │
│  3. Загружаем последние 15 сообщений + summary  │
│  4. Классифицируем намерение (gpt-4o-mini)      │
│  5. Вызываем gpt-4o с tools                     │
│  6. Если есть tool_calls — выполняем            │
│  7. Сохраняем сообщения в БД                    │
│  8. Отправляем ответ в Telegram                 │
│  9. Проверяем триггеры передачи менеджеру       │
└─────────────────────────────────────────────────┘
    ↓                              ↓
Supabase Postgres            Bitrix24 REST API
(users, messages,            (создаём лид/сделку,
 properties, leads,           передаём диалог)
 viewings, embeddings)
```

## Контекст диалога

Хранить контекст полностью дорого и плохо влияет на качество ответов. Стратегия:

1. **Последние 15 сообщений** — отдаются в OpenAI как есть
2. **Summary старше 15 сообщений** — генерируется через `gpt-4o-mini` каждые 10 новых сообщений, хранится в `users.context_summary`
3. **Извлечённые данные лида** — хранятся структурированно в `leads` и подмешиваются в системный промпт как факты

## RAG для объектов и базы знаний

Объекты недвижимости + база знаний (ипотечные программы, ЖК, FAQ) векторизуются через `text-embedding-3-small` и хранятся в `pgvector` колонке.

**Гибридный поиск:**
1. Фильтры по жёстким критериям (цена, район, комнаты) — SQL WHERE
2. Семантический ranking — `<=>` оператор pgvector по embedding запроса
3. Top-5 объектов передаются в gpt-4o как контекст

## Передача менеджеру (hand-off)

Триггеры для эскалации:
- Клиент явно просит человека («хочу с менеджером», «соедините»)
- Готовность к просмотру + бюджет выше порога
- Жалоба или негатив (детектится классификатором)
- Юридический вопрос (не отвечаем сами)
- 3+ сообщения подряд без прогресса

При эскалации:
1. AI генерирует summary диалога
2. Создаётся сделка в Битрикс24 с этим summary
3. Менеджеру приходит уведомление
4. Бот говорит клиенту, что менеджер свяжется в ближайшие N минут
5. Бот **переходит в режим silent** для этого чата (не отвечает, пока менеджер не закроет диалог)

## Follow-up последовательности

APScheduler крутит задачу раз в час и проверяет:
- Лиды без активности 24 часа → отправить «Подобрали ещё варианты, посмотрите?»
- Лиды без активности 3 дня → отправить «Изменились ли требования?»
- Лиды без активности 7 дней → отправить «Закрываем заявку или продолжаем поиск?»

Каждый follow-up логируется. Если клиент не отвечает на 3-й — лид помечается `cold` и больше не пишем.

## Безопасность

- Telegram webhook защищён secret_token (валидация в middleware)
- Supabase Service Role Key только на бэкенде, никогда в коде клиента
- RLS на Supabase для всех таблиц
- Bitrix24 webhook URL хранится в env, ротация раз в квартал
- OpenAI API key через переменные окружения, не в коде
- Все ПД (телефон, имя) шифруются на уровне приложения перед записью в БД (опционально, если есть требование)
